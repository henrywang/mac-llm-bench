Subject: API rate limiting rollout — plan & blockers

Hi team,

Following last week's incident where a single client hammered the /search endpoint and took down
the cluster for 40 minutes, we need to ship rate limiting before end of month. I want to align
on the plan today.

Current proposal: per-client token bucket at the API gateway layer, 1000 req/min default,
configurable per tier. Redis-backed counters with a 60s sliding window.

Blockers I'm aware of:
- We don't have a Redis cluster in prod yet (only staging). Infrastructure ticket is open.
- Client SDK needs to handle 429 responses gracefully. Right now it retries immediately,
  which would make the problem worse.
- Docs for the new headers (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset)
  need to be written before we can notify enterprise customers.

Let me know your thoughts.
Sarah
